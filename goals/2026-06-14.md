# Tako Feature Plan: Meshed takod Operator Surface

Date: 2026-06-14

This plan captures the gap review against Uncloud and turns it into a focused
implementation path for Tako. The product direction stays narrow: one meshed
`takod` runtime, app/stage isolation, simple config, and practical operator
commands.

## Current Tako Baseline

Tako already has:

- One runtime path: `tako CLI -> takod -> local Docker`.
- Single-node as a one-node mesh.
- App/stage isolation for containers, networks, proxy files, state, leases, env
  bundles, and generated volumes.
- Remote leases for laptop and CI safety.
- Replicated desired state, actual state, deployment history, and state repair.
- Encrypted env push/pull for cloned computers and CI.
- Scoped cleanup that preserves shared `takod` and proxy runtime.
- Basic service logs, metrics, scale, rollback, destroy, and remove commands.
- Deployment pattern examples for common app shapes.

The missing surface is mostly day-2 operations, scheduler intelligence, and
runtime polish.

## High Priority Gaps

### 1. Exec and One-Off Run

Add operator commands for debugging, admin tasks, and migrations:

```text
tako exec SERVICE [COMMAND...]
tako exec --replica SLOT SERVICE [COMMAND...]
tako run SERVICE [COMMAND...] --one-off
tako run SERVICE [COMMAND...] --one-off --rm
```

Requirements:

- Default to an interactive shell when no command is provided.
- Support non-TTY mode for pipes and CI.
- Select a replica deterministically unless a replica or node is specified.
- Return the command exit code.
- Stream stdout/stderr without storing secrets in state.
- Use `takod` socket APIs, not direct Docker SSH mutation.

### 2. Local Port Forwarding

Add local forwarding for private/internal services:

```text
tako proxy SERVICE REMOTE_PORT
tako proxy SERVICE LOCAL_PORT:REMOTE_PORT
tako proxy --server node-a SERVICE 5432
```

Requirements:

- Bind locally on `127.0.0.1`.
- Select a healthy target replica.
- Work for services without public `proxy`.
- Prefer mesh/private routes.
- Print a clear local URL/address.
- Exit cleanly on connection errors or Ctrl-C.

### 3. Pre-Deploy and Post-Deploy Hooks

Add first-class hooks instead of relying on service command chaining:

```yaml
hooks:
  preDeploy:
    command: npm run migrate
    timeout: 5m
  postDeploy:
    command: npm run warm-cache
    timeout: 2m
```

Requirements:

- Run hook containers from the service image.
- Support env, secrets, user, working dir, and timeout.
- Fail deploy on non-zero exit.
- Keep failed hook logs inspectable.
- Clean old successful hook containers.
- Run hooks once per service deployment by default, with future room for
  per-node hooks.

### 4. Rolling Deploys

Move beyond recreate-only deployment.

```yaml
deploy:
  strategy: rolling
  order: start-first
  maxUnavailable: 0
  monitor: 30s
```

Requirements:

- Start-first when ports allow it.
- Stop-first fallback when host ports conflict.
- Health-gate each replacement.
- Preserve old containers until the new one is healthy.
- Roll back or keep the old version serving if the new version fails.
- Support `maxUnavailable` for multi-replica services.

### 5. Volume-Aware Scheduling

Make node-local volume behavior explicit and safe.

Requirements:

- If a named volume already exists on a node, schedule the service there unless
  placement explicitly says otherwise.
- Co-locate services sharing a named volume.
- For global services, ensure required volumes exist on every eligible node.
- Reject ambiguous multi-node writable volume setups unless the service declares
  external/application-level replication.
- Explain placement failures with the service, volume, and eligible nodes.
- Keep generated volume names app/stage scoped.

### 6. Rich Ports Model

Keep `port` as shorthand, but add `ports` for serious services.

```yaml
ports:
  - name: http
    target: 3000
    protocol: http
    proxy:
      domain: app.example.com
  - name: metrics
    target: 9090
    protocol: tcp
    internal: true
  - name: dns
    target: 5353
    published: 53
    protocol: udp
    mode: host
    hostIP: 10.0.0.0/24
```

Requirements:

- Multiple named ports per service.
- TCP, UDP, HTTP, and HTTPS protocols.
- Ingress/proxy mode and host-bind mode.
- Optional bind IP or CIDR for host mode.
- Clear conflict detection against existing Docker bindings and Tako
  allocations.
- Backward-compatible rendering from simple `port`.

### 7. Internal Service Discovery

Add a private discovery story for service-to-service communication.

Initial scope:

```text
SERVICE.ENV.PROJECT.tako.internal
SERVICE.tako.internal within the same app/stage network
```

Requirements:

- Return healthy internal service endpoints.
- Prefer local replicas when available.
- Support round-robin answers.
- Keep services private unless explicitly exported.
- Later extension: SRV records for named ports.

### 8. Prometheus-Compatible Metrics

Keep `tako metrics`, but expose scrape-friendly metrics from `takod`.

Requirements:

- `takod` exposes `/metrics` on a mesh/local-only listener.
- Include node health, disk, memory, load, service replica counts, deploy state,
  proxy upstream health, and lease state.
- Do not expose metrics publicly by default.
- Add docs for Prometheus scraping through the mesh or `tako proxy`.

### 9. Image and Build Ergonomics

Improve image distribution and platform support.

Requirements:

- Build cache support through `cache-from`/`cache-to` or a registry cache.
- `build.platform` and buildx support.
- Detect target node architecture and warn on mismatch.
- Support private registry auth paths.
- Improve local Docker compatibility for rootless Docker, Docker Desktop,
  Colima, and other VM-backed engines.
- Avoid transferring full image archives when only layers changed.

### 10. Operator UX Commands

Add commands that make the mesh easier to inspect and repair:

```text
tako inspect [SERVICE]
tako node logs [NODE]
tako mesh rtt
tako volume ls
tako volume backup SERVICE VOLUME
tako volume restore SERVICE VOLUME BACKUP_ID
tako image ls
tako image prune
```

Requirements:

- Prefer typed `takod` APIs.
- Use app/stage scoping by default.
- Include node/service/replica IDs in output.
- Add dynamic shell completion for services, nodes, volumes, images, and stages.

## Completion Audit

The current single-path meshed `takod` model covers the practical P0-P2
operator surface in this plan: exec, one-off run, proxy, hooks, safe volume
placement, rolling deploys, explicit ports, private discovery, Prometheus
metrics, node logs, mesh RTT, build cache, platform checks, private registry
auth, dynamic completions, resource cleanup, volume backup/restore, inspect,
and image prune.

Remaining follow-ups are explicitly future work, not blockers for the current
CLI migration: exported cross-project discovery records, an optional mesh
DNS/SRV listener, per-platform multi-arch manifest publishing, deeper
rootless/Docker Desktop/Colima compatibility proof, and peer image
distribution that avoids full archive transfer when only layers changed.

## Non-Goals For Now

- Do not turn Tako into a full Docker Compose compatibility layer.
- Do not replace the current replicated `takod` state model with Corrosion or a
  separate distributed database unless the simpler model proves insufficient.
- Do not add a managed DNS product until the local/private/custom-domain path is
  solid.
- Do not add Kubernetes-style abstractions unless they directly simplify the
  single Tako workflow.

## Implementation Order

Start with the features that reduce operator friction without reshaping the
whole runtime:

1. Add `takod` APIs for exec, one-off run, and local proxy target discovery.
2. Add `tako exec`, `tako run --one-off`, and `tako proxy`.
3. Add hook config and hook execution in the deploy plan.
4. Add volume-aware placement checks before runtime mutation.
5. Add rolling deploy strategy on top of the existing health checks.
6. Introduce `ports[]` while preserving `port` (done).
7. Add internal discovery and Prometheus metrics once the runtime model is
   stable.

## Phase Gate

Every phase must be tested against the playground before it is marked done:

- Run focused unit/integration tests for the changed code.
- Build the local CLI and, when `takod` changes, a Linux amd64 binary.
- Refresh setup on the playground node when server-side code changes.
- Deploy an isolated app/stage fixture that does not collide with existing
  playground projects.
- Exercise the new feature plus the surrounding lifecycle commands it depends
  on: `deploy`, `ps`, `logs`, `metrics`, `state pull`, `drift` or `doctor`,
  and cleanup.
- Verify cleanup by label/name over SSH so unrelated projects, `takod`, and
  `tako-proxy` are preserved.
